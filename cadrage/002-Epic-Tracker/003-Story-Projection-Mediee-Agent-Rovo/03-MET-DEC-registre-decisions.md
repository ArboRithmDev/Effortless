## 🏛️ Decision List

### 1. `DEC-01` : Révision DEC-03 : projection médiée par l'agent (Rovo), pas REST serveur
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : DEC-03 (STO-TRACKER-01) imposait un adapter REST côté serveur. Coût : token Atlassian à gérer (expirant), exposé en transcript. Le serveur ne peut pas appeler les MCP.
* **Decision** : La projection Jira est médiée par l'agent : le serveur planifie (enqueue dans l'outbox), l'agent exécute via le connecteur Rovo MCP, puis ack les refs. Le serveur ne détient aucun credential et ne fait aucun I/O réseau.
* **Consequences** :
  - Zero token a gerer cote utilisateur
  - Aucun secret dans le serveur ni le transcript
  - Depend du connecteur Rovo cote agent/CLI

### 2. `DEC-02` : Retrait du JiraClient REST et du token
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : L'ancien chemin REST+token devient inutile et indésirable (secret). Ben : Rovo seul.
* **Decision** : Suppression de ports/adapters/jira_client.py::JiraClient (REST), de build_jira_tracker (creds env) et de toute lecture JIRA_*. FakeJiraClient est conservé pour les tests (il simule l'exécuteur agent).
* **Consequences** :
  - Plus aucun mode headless sans agent Rovo
  - Surface d'attaque/secret nulle
  - Tests restent hermetiques via FakeJiraClient

### 3. `DEC-03` : Outbox SyncJournal = transport ; tools pending/ack
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Il faut un canal serveur→agent→serveur pour les ops et leurs résultats. SyncJournal (outbox rejouable idempotent) existe déjà depuis STO-TRACKER-01.
* **Decision** : QueueTracker.create enqueue une op dans le SyncJournal. effortless_tracker_pending renvoie les ops en attente (plan JSON, parent-avant-enfant). effortless_tracker_ack reçoit la map {local_id:{tracker_id,tracker_url}}, persiste l'identité et marque l'outbox joué (idempotent).
* **Consequences** :
  - Reutilise un acquis (zero dette nouvelle)
  - Rejouabilite/idempotence heritees
  - Le flush peut etre repris apres interruption

### 4. `DEC-04` : QueueTracker : create enqueue, ref placeholder local:N
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : La clé Jira réelle n'existe pas à l'enqueue ; le scaffolder a besoin d'une ref parent pour câbler les enfants.
* **Decision** : QueueTracker.create retourne TrackerRef(tracker_id="local:N"). Le scaffolder (inchangé) l'utilise comme parent_ref ; l'op enfant porte parent_local_id="local:N". L'agent résout local:N → clé réelle à l'exécution et renvoie la map via ack.
* **Consequences** :
  - scaffolder reutilise tel quel (agnostique)
  - Liens parent encodes par id local
  - L'agent fait le tri topologique a l'execution

---

### 5. `DEC-05` : Disclaimer Rovo + précondition agent
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le serveur ne peut pas détecter si l'agent a le connecteur Rovo. Sans Rovo, le flush Jira est impossible.
* **Decision** : Chaque outil tracker préfixe sa sortie d'un disclaimer : « projection médiée — nécessite le connecteur Atlassian Rovo MCP déclaré dans la CLI/App ». Une consigne dans CLAUDE.md/skill impose à l'agent de vérifier la présence de Rovo avant pending/flush.
* **Consequences** :
  - Utilisateur sans Rovo prevenu explicitement
  - Pas de detection serveur (discipline agent)
  - Doc d'install Rovo a referencer

### 6. `DEC-06` : Idempotence : ScaffoldState locale primaire, reconcile agent optionnel
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le serveur ne peut pas chercher dans Jira pour dédupliquer.
* **Decision** : Garde primaire = ScaffoldState locale (déterministe, zéro réseau). Garde renforcée optionnelle (hors MVP) = l'agent fait searchJiraIssuesUsingJql du label effortless-scaffold:&lt;zone&gt; avant flush ; si présent, skip.
* **Consequences** :
  - Re-run deterministe sans reseau
  - Reconcile Jira-as-truth reste possible cote agent
  - Garde locale suffit au MVP

---

### 7. `DEC-07` : Flush Rovo : forcer le type d'issue par id (sous-tâches)
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : E-execute
* **Context** : Validation live : Rovo createJiraIssue a rejeté issueTypeName="Sous-tâche" (« type de ticket non valide ») alors que l'API REST acceptait l'id 10095. Les noms de type ne suffisent pas côté Rovo pour les sous-tâches.
* **Decision** : Procédure de flush agent : avant d'exécuter le plan, résoudre la taxonomie via getJiraProjectIssueTypesMetadata (nom→id) et passer le type par id dans additional_fields={"issuetype":{"id":...}} sur chaque createJiraIssue. Robuste pour tous les niveaux, indispensable pour les sous-tâches. Suivi possible : enrichir l'op du plan avec issue_type_id (nécessite discover médié).
* **Consequences** :
  - Flush deterministe (id != nom)
  - Sous-taches creees de maniere fiable
  - Discover medie (name->id) devient un prerequis du flush
