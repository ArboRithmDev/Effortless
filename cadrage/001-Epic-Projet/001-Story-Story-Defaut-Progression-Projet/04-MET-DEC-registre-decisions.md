## 🏛️ Decision List

### 1. `DEC-01` : Modèle de domaine natif Epic/Story/Task (local-first)
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Trou structurel entre projet et tâche ; besoin d'une couche de décomposition indépendante de tout tracker.
* **Decision** : Epic/Story/Task sont des entités de domaine de premier rang persistées localement, que Jira soit branché ou non.
* **Consequences** :
  - Le cœur reste agnostique du tracker
  - Jira devient une projection optionnelle via adapter
  - Identifiants TSK- en repli local sans tracker
* **Rejected Alternatives** :
  - Modèle piloté par Jira (couplage fort, non portable)

---

### 2. `DEC-02` : OPALE est un processus de Story
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le projet portait un singleton current_phase global et une liste de tâches à plat, traitant le projet comme une itération unique.
* **Decision** : La position OPALE est portée par chaque Story ; suppression du current_phase global. Le projet suit la Story courante et l'arbre Epic/Story.
* **Consequences** :
  - Modèle fractal : N petites passes OPALE composables
  - ProjectState.current_phase migre vers Story.current_phase
  - Docs de cadrage deviennent Story-scopés
* **Rejected Alternatives** :
  - Phase OPALE globale au projet (ne scale pas, pas de parallélisme)

---

### 3. `DEC-03` : Vocabulaire à trois axes orthogonaux
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le terme « phase » désignait deux concepts (OPALE et itération de développement), brouillant les instructions agent.
* **Decision** : « phase » = OPALE uniquement ; « Story » = itération de développement ; « step »/« état » = boucle d'exécution. Trois axes orthogonaux.
* **Consequences** :
  - Purge documentaire : « phase » au sens itération bascule en « Story »
  - Instructions agent désambiguïsées

---

### 4. `DEC-04` : Synchronisation Jira : push + import ponctuel
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le bidirectionnel continu impose une résolution de conflits coûteuse.
* **Decision** : Effortless = source de vérité. Push vers Jira (création/transition/worklog) ; pull uniquement à l'import initial.
* **Consequences** :
  - Pas de réconciliation continue
  - Issues humaines importées restent read-mostly
* **Rejected Alternatives** :
  - Synchronisation bidirectionnelle continue

---

### 5. `DEC-05` : Worklog et commentaire au niveau Sous-tâche
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Besoin de tracer finement le temps et les interventions par tâche, IFX logge au niveau Story.
* **Decision** : Worklog + commentaire attachés à la Sous-tâche (= tâche Effortless), avec rollup automatique vers la Story parente.
* **Consequences** :
  - Traçabilité fine par tâche
  - Cumul remonté à la Story
* **Rejected Alternatives** :
  - Worklog au niveau Story uniquement (moins granulaire)

---

### 6. `DEC-06` : Intégration Xray dès la v1
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : L'instance Jira de référence (IFX) expose Test/Test Set/Test Plan (Xray).
* **Decision** : La phase de recette OPALE crée des Tests Xray rattachés à la Story et trace la couverture dans Jira.
* **Consequences** :
  - Couverture de test tracée dans Jira
  - Surface d'intégration élargie dès v1
* **Rejected Alternatives** :
  - Xray hors scope v1 (recette locale seule)

---

### 7. `DEC-07` : Estimation via time-tracking en heures
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le projet de référence est en cycle en V et utilise le time-tracking, pas les Story Points.
* **Decision** : L'estimation est poussée à la création via le champ timeoriginalestimate (heures).
* **Consequences** :
  - Cohérent avec le worklog en heures
  - Pas de champ Story Points en v1
* **Rejected Alternatives** :
  - Story Points (custom field, non utilisé en cycle en V)

---

### 8. `DEC-08` : Cardinalité 1 Epic : N Stories
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Observé dans IFX (Epic IFX-7 porte 5 Stories).
* **Decision** : Un Epic accumule N Stories ; une Story = une itération OPALE.
* **Consequences** :
  - Un Epic vit plusieurs itérations dans le temps
  - Tag [ZONE] hérité de l'Epic vers ses Stories

---

### 9. `DEC-09` : Onboarding brownfield par niveaux d'intégration N0–N3
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : La reprise d'un projet non cadré doit offrir plusieurs niveaux d'engagement, paliers stables.
* **Decision** : Échelle N0 Observation / N1 Cadrage / N2 Refacto cadrée / N3 Adoption. La migration auto-génère ses propres Epics/Stories [MIGRATION].
* **Consequences** :
  - Adoption incrémentale, arrêt possible à chaque palier
  - Migration uniforme avec le modèle fractal
* **Rejected Alternatives** :
  - Adoption tout-ou-rien

---

### 10. `DEC-10` : Pattern store enrichissable à format standard
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le pattern cible par stack doit être réutilisable, proposé et extensible.
* **Decision** : Un store de patterns de base, hydraté par recherche web approfondie par stack, enrichissable ; un modèle à remplir (gabarit standard) sert de format de descripteur pour tout pattern fourni.
* **Consequences** :
  - Catalogue extensible projet/framework/utilisateur
  - Synergie avec le framework maison de cadrage technique multiplateforme

---

### 11. `DEC-11` : « Sans douleur » = philosophie d'orchestration totale
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Risque de sur-promesse technique sur la migration.
* **Decision** : On accepte autant d'étapes que nécessaire ; Effortless décompose et orchestre, l'utilisateur n'orchestre rien (« Effortless s'occupe de tout »).
* **Consequences** :
  - Migration découpée en petites passes nombreuses
  - Charge d'orchestration portée par l'outil

---

### 12. `DEC-12` : Dérivation backlog auto avec arbitrage utilisateur
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : La structure découverte (clusters/smells/frontières) doit produire un backlog sans imposer un découpage rigide.
* **Decision** : La dérivation structure→Epics/Stories est proposée automatiquement (heuristique) ; l'utilisateur arbitre le découpage final.
* **Consequences** :
  - Cluster/frontière → Epic, smell/use-case → Story, élément → Sous-tâche
  - Décision finale humaine

---

### 13. `DEC-13` : Idempotence du re-scan avec arbitrage
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : L'onboarding doit être rejouable sans créer de doublons.
* **Decision** : Idempotence visée ; en cas de divergence entre re-scan et backlog existant, l'arbitrage revient à l'utilisateur (pas de réconciliation magique).
* **Consequences** :
  - Re-scan sûr
  - Divergences exposées plutôt que fusionnées silencieusement

---

### 14. `DEC-14` : Migration isolée sur branche dédiée
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Une refacto cadrée doit rester réversible et ne pas casser la branche principale.
* **Decision** : Toute migration se fait sur une branche dédiée ; merge vers la principale sur arbitrage de l'utilisateur. Rollback = branche + un commit par Story.
* **Consequences** :
  - Réversibilité garantie
  - Worktree d'isolation envisageable pour Stories parallèles

---

### 15. `DEC-15` : Coexistence avec un Jira existant
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Un legacy peut être déjà câblé à un projet Jira avec des issues humaines.
* **Decision** : Import read-mostly des issues existantes (possédées par l'humain) ; le backlog de migration se greffe en Epic [MIGRATION] dans le même projet. Effortless ne possède que ce qu'il crée.
* **Consequences** :
  - Reporting et temps unifiés
  - Pas d'écrasement des issues humaines
* **Rejected Alternatives** :
  - Projet Jira séparé pour la migration (reporting scindé) — offert en option

---

### 16. `DEC-16` : Filet de sécurité dimensionné au ROI
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le legacy non testé rend risquée toute refacto ; un golden master systématique serait coûteux.
* **Decision** : Le filet de sécurité (tests de caractérisation) est dimensionné au ROI : effort proportionné au risque et à la valeur, pas systématique.
* **Consequences** :
  - Effort de test ciblé sur les zones à risque/valeur
  - Heuristique de scoring ROI à spécifier (Articuler)

---

### 17. `DEC-17` : Adaptation des outils de migration existants
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : migration_planner (variantes M-*) et migrate_init/apply existent dans le modèle global actuel.
* **Decision** : Ils sont adaptés au modèle Story-scopé : la variante OPALE migration devient le processus que run chaque Story de refacto ; le ladder est la couche au-dessus.
* **Consequences** :
  - Réutilisation de l'existant
  - Refonte de migration_planner pour cibler une Story

---

### 18. `DEC-18` : Évolution de wlReForge vers le modèle Effortless
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : wlReForge produit clusters/smells/frontières mais pas dans le format Epic/Story/Task.
* **Decision** : wlReForge évolue pour que sa production matche le modèle Effortless ; wlReForge est lui-même un candidat à la migration (dogfood).
* **Consequences** :
  - Contrat de sortie wlReForge → backlog à définir (Articuler)
  - wlReForge premier client du ladder de migration
