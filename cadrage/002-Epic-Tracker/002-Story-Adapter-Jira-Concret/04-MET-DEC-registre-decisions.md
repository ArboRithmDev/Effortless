## 🏛️ Decision List

### 1. `DEC-01` : Scaffold = service domaine, pas une 6e op du Port
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le brief proposait d'ajouter scaffold comme 6e opération du Tracker Protocol. Les 5 ops actuelles sont des primitives atomiques agnostiques ; le scaffold est un workflow composite (boucle template → N×create + câblage parent), lui-même agnostique.
* **Decision** : Le scaffold reste HORS du Port. Il est implémenté dans un service domaine scaffold_project_from_template() au-dessus du Port, qui n'appelle que create(). Le Port garde ses 5 primitives.
* **Consequences** :
  - Aucun futur adapter ne reduplique la boucle de scaffold
  - Port minimal et stable
  - Le composer est testable independamment de tout tracker concret

### 2. `DEC-02` : Périmètre MVP STO-TRACKER-02 = slice vertical scaffold
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le brief mêlait adapter Jira complet (5 ops) et feature scaffold. Livrer les 5 ops avant tout usage rallonge la livraison sans preuve bout-en-bout.
* **Decision** : MVP = register_adapter("jira") + discover_taxonomy + create + composer de scaffold, prouvé end-to-end par le scaffold du template [PROJET] dans EFL. transition/log_work/import/Xray repoussés en stories suivantes du backlog M2.
* **Consequences** :
  - Preuve concrete rapide que le contrat Port fonctionne contre une vraie instance
  - Les ops non-MVP restent no-op/NotImplementedError documentees
  - Backlog M2 etale en stories successives

### 3. `DEC-03` : Adapter Jira piloté par un client REST injecté
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le process serveur Python ne peut pas appeler les MCP Atlassian (capacité côté agent). La projection (task_add/update) s'exécute dans le serveur, qui doit donc joindre Jira lui-même.
* **Decision** : JiraTracker dépend d'un JiraClient (REST Jira Cloud v3, auth token depuis cfg/env) injecté à la construction. Tout I/O réseau est confiné dans JiraClient. Les tests injectent un FakeJiraClient (fixtures), zéro réseau. JiraTracker et le scaffolder restent purs.
* **Consequences** :
  - Suite pytest hermetique (aucun appel reseau reel)
  - Frontiere I/O nette et mockable
  - Credentials Jira a externaliser dans settings.tracker/env

### 4. `DEC-04` : Template [PROJET] externalisé en JSON, init depuis IFX-1
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Q-01 cadrage : template figé en code vs découvert vs externalisé. Figé fige la structure ; découvert depuis IFX dépend d'IFX accessible.
* **Decision** : Template décrit dans templates/jira_project_scaffold.json, versionné dans le repo Effortless, initialisé depuis l'observation d'IFX-1 (1 Epic + 3 Stories + 2 sous-tâches). Modifiable sans toucher au code ni dépendre d'IFX à l'exécution.
* **Consequences** :
  - Structure du scaffold modifiable par config
  - Pas de dependance runtime a IFX
  - Le template suit le versioning du repo

### 5. `DEC-05` : Cible d'écriture = EFL ; IFX en lecture seule
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le scaffold crée de vraies issues. Tester contre IFX (projet réel actif) polluerait des données métier.
* **Decision** : Toutes les écritures de test visent EFL (projet vide dédié, company-managed, types 1:1 avec IFX). IFX reste strictement en lecture seule (source du template + référence taxonomie).
* **Consequences** :
  - Aucune ecriture sur des donnees metier reelles
  - EFL = bac a sable reproductible
  - Le mapping de types vaut tel quel sur EFL (ids identiques)

### 6. `DEC-06` : Idempotence du scaffold : label Jira + tracker_id local
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Un re-run naïf du scaffold dupliquerait l'arbre [PROJET] côté Jira.
* **Decision** : Double garde : (1) label effortless-scaffold:&lt;zone&gt; posé sur l'Epic racine ; le composer interroge Jira et skip si présent (source de vérité = Jira). (2) Court-circuit local si tracker_id déjà persisté dans .effortless/. Le label prime en cas de divergence.
* **Consequences** :
  - Re-run sur projet deja amorce = no-op
  - Robuste a la perte d'etat local (verite cote Jira)
  - 1 requete search de garde avant scaffold

### 7. `DEC-07` : Scaffold fidèle au template : non affecté, structure nue
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Le template IFX-1 [PROJET] est entièrement non affecté (assignee null) et sans estimation (timeoriginalestimate null). Brief : « non affectée telle quel ».
* **Decision** : Le composer pose assignee=null sur tous les nœuds et ne renseigne aucune estimation. Affectation et estimation se font ultérieurement (manuellement côté Jira, ou via create() de Task réelles pour l'estimation).
* **Consequences** :
  - Reproduction fidele du template observe
  - Scaffold = structure pure, sans donnees de pilotage
  - Projection d'estimation traitee hors scaffold

---

### 8. `DEC-08` : Taxonomie et transitions résolues dynamiquement
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : P-cadrage
* **Context** : Les ids de types et surtout les transitionId sont variables par instance/projet Jira. Les coder en dur casserait l'agnosticité et la portabilité.
* **Decision** : discover_taxonomy est appelé systématiquement au couplage. Les transitionId sont résolus à l'exécution via get_transitions, jamais codés en dur. Un mapping est proposé si la taxonomie découverte diverge du mapping canonique attendu.
* **Consequences** :
  - Adapter portable sur d'autres instances Jira
  - Aucune fragilite sur des ids codes en dur
  - discover_taxonomy est un prerequis du couplage
