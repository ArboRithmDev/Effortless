## 💬 Question Details

### Q-01 : Comment l'agent identifie-t-il les ops à exécuter puis à acker (corrélation plan ↔ refs créées) ?
* **Context** : En mode médié, create() enqueue avant que la clé Jira existe. Le plan doit porter des id locaux stables que l'agent résout en clés réelles puis renvoie. Le parent est référencé par id local (la vraie clé n'existe pas à l'enqueue).
* **Impact** : **Structuring**
* **LLM Suggestion** : QueueTracker.create retourne une ref placeholder TrackerRef(tracker_id="local:N"). Le scaffolder l'utilise comme parent_ref → l'op enfant porte parent_local_id="local:N". pending renvoie la liste d'ops JSON (avec local_id). ack reçoit {local_id: {tracker_id, tracker_url}}.
* **Status** : Resolved
* **User Answer** : QueueTracker.create retourne TrackerRef(tracker_id="local:N", tracker_url="") (placeholder à id local). Le scaffolder l'utilise comme parent_ref des enfants → l'op enfant porte parent_local_id="local:N". effortless_tracker_pending renvoie la liste d'ops JSON (op, local_id, level, title, parent_local_id, labels). effortless_tracker_ack reçoit {local_id: {tracker_id, tracker_url}}, persiste l'identité et marque les entrées outbox jouées.
* **Date Resolved** : 2026-06-30
