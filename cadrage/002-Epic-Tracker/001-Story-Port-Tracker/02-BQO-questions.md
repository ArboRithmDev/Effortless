* **Impact** : **Structuring**
* **LLM Suggestion** : typing.Protocol (duck-typing, mock trivial, pas d'héritage imposé) + un adapter null par défaut.
* **Status** : Resolved
* **User Answer** : Abstraction agnostique unique exposant un métier COMMUN aux 4 niveaux d'objet (Projet, Epic, Story, Task→Sub-Task). Forme technique : typing.Protocol (agnostique, mockable, pas d'héritage imposé). Un adapter null par défaut.

### Q-03 : Stockage du mapping ID local ↔ clé tracker : champ tracker_key sur epic/story/task.json, ou table séparée .effortless/tracker_map.json ?
* **Context** : Quand couplé, l'ID Effortless EST la clé d'issue. Mais avant couplage les entités existent sans clé ; et un même projet peut être (re)couplé. Décidé au CORE : identité partagée quand couplé.
* **Impact** : **Structuring**
* **LLM Suggestion** : Champ tracker_key Optional sur chaque entité (back-compat, co-localisé, diff-friendly). Table séparée seulement si multi-tracker simultané requis (pas le cas).
* **Status** : Resolved
* **User Answer** : NON optionnel. Chaque objet (epic/story/task) porte tracker_id + tracker_url de l'issue Jira correspondante ; le niveau projet porte l'ID + URL de l'espace. Champs canoniques standards (pas Optional/afterthought). Justification : coût de persistance nul, réduit structurellement l'effort, et fonde un métier commun aux 4 adaptateurs (Projet, Epic, Story, Sub-Task).
* **Date Resolved** : 2026-06-30

### Q-05 : Politique d'erreur / hors-ligne du port : fail-fast (remonter l'erreur) ou file d'attente + retry idempotent ?
* **Context** : Le push peut échouer (réseau, droits, tracker indisponible). Effortless reste fonctionnel sans tracker. Impact faible sur le CONTRAT du port, tranchable plus tard.
* **Impact** : **Minor**
* **LLM Suggestion** : Fail-fast v1 : l'opération locale réussit toujours, la projection tracker est best-effort et journalise l'échec sans bloquer. File/retry = amélioration ultérieure.
* **Status** : Resolved
* **User Answer** : Pas de fail-fast oubliant l'échec. Si Jira est injoignable, on tient un JOURNAL des éléments à synchroniser, rejoué dès le dialogue rétabli. Renforcé : chaque élément en attente est une MIGRATION que l'on joue et que l'on flag comme jouée avec un timestamp (outbox idempotent, rejouable, traçable).
* **Date Resolved** : 2026-06-30
