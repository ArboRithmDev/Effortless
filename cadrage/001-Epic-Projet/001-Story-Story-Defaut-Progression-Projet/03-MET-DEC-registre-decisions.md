### 19. `DEC-19` : Dogfood M1-F1: loop recette runner crashes on strict utf-8 under Windows
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : L-plan
* **Context** : First Recette pass (TSK-L-06) returned 'Failed to run tests: can only concatenate str (not NoneType) to str' while pytest passes (21 passed, exit 0 manually). Root cause: subprocess.run(shell=True, text=True, encoding=utf-8) with no errors=. Under shell=True (cmd.exe), an OEM/cp850 byte (0x82 = e-acute in the French cmd.exe error) makes the reader thread raise UnicodeDecodeError, so res.stderr becomes None and the concat res.stdout + newline + res.stderr crashes. Real exit code and message are masked.
* **Decision** : Recette runner must decode tolerantly: subprocess.run(encoding=utf-8, errors=replace) or capture bytes then decode utf-8/replace. Guard against None stdout/stderr before any concat.
* **Consequences** :
  - Runner no longer masks real failures behind a UnicodeDecodeError.
  - Story [CORE]: add errors=replace and None-guard in session_loop.py.
  - Non-blocking once test_command uses backslash exe path.

### 20. `DEC-20` : Dogfood M1-F3: loop auto-commit blocked by own pre-commit anti-drift hook
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : L-plan
* **Context** : On Recette success, session_loop sets the active task to Done, THEN runs git add + git commit. The project pre-commit hook runs cli/main.py --drift-check-strict, which blocks any commit when files are modified but NO task is in status Doing. Since the loop already flipped the task to Done, no Doing task remains, so the hook exits 1 and the commit fails. The loop swallows this as 'already committed?'. Result: delivered work is never committed; the uncommitted file then trips the loop's internal check_project_drift on the NEXT task (file outside active task = drift), cascading to block every subsequent task.
* **Decision** : Fix ordering: the loop must commit while the task is still Doing (commit first, then mark Done), OR pass the just-completed task id to the drift check so a fresh-Done task is accepted. Interim: the git pre-commit hook is redundant with the loop's internal check_project_drift during autonomous runs; commits use --no-verify until the ordering fix ships.
* **Consequences** :
  - Hard blocker: without fix, the autonomous loop cannot persist any work on this setup.
  - Cascade: uncommitted delivery trips next-task drift check.
  - Story [CORE]: reorder commit-before-Done (or pass task id to drift check) in session_loop.py.
  - Operational workaround now: manual git commit --no-verify after each loop delivery.

### 22. `DEC-22` : Storage topology M1: nested per-Epic with Story registry, zone/story-scoped IDs
* **Status** : Accepted
* **Date** : 2026-06-30
* **Phase** : L-plan
* **Context** : Current .effortless is flat: decisions/ questions/ tasks/ are global with a phase-letter ID prefix (TSK-L-NN). This works only because there is a single global phase; it does not follow the fractal Epic>Story>Task model. Gap surfaced by Ben during L-19 prep: the JSON store nomenclature was never designed to grow organically with the model. The flat epics/ stories/ dirs added by L-09 are insufficient.
* **Decision** : Nested-per-Epic layout. .effortless/epics/EPIC-<ZONE>/ holds epic.json (carrying a stories[] registry = the Story registry at the Epic root) plus a stories/ dir. Each STO-<ZONE>-NN/ folder is a self-contained OPALE workspace: story.json + tasks/ + decisions/ + questions/ (+ optional cadrage/). state.json keeps active_story_id and active_epic_id; no current_phase after final removal. IDs are zone/story-scoped and readable: Epic = EPIC-<ZONE>; Story = STO-<ZONE>-NN (sequence per Epic); Task/Decision/Question = TSK-NN / DEC-NN / Q-NN sequenced WITHIN the owning Story (restart per Story). Path resolution and ID generators become Story-scoped (resolve via active Epic+Story).
* **Consequences** :
  - Supersedes the flat epics/stories paths added in L-09.
  - New foundational task before L-19: fractal path-resolution + ID-generator layer in server.py.
  - get_active_story must locate the story under epics/*/stories/*; helpers load/save/next_id operate within the active Story dir.
  - Migrator (L-15) builds EPIC-PROJET/stories/STO-PROJET-01 and moves the existing global cadrage data (DEC-*, Q-*, TSK-*) into it.
  - L-19 task_add prefixes and reads now resolve via the active Story dir.
